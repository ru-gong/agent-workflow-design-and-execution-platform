DAG 工作流、任务编排工具与对话协作的区别

这份说明用于记录：为什么这个工具底层叫 DAG 工作流，以及它和直接对话协作分别适合什么场景。

为什么叫 DAG 工作流

我叫它 DAG 工作流，主要是从实现模型说的，不一定是给用户看的产品名。

DAG = Directed Acyclic Graph，有向无环图。放到这个工具里就是：

  • 节点：一个子任务，比如“需求拆解”“实现页面”“运行测试”“人工确认”“汇总交付”。
  • 边：依赖关系，比如“测试”必须等“实现”完成。
  • 无环：不能出现 A 等 B、B 又等 A 的死循环。
  • 好处：可以自动判断谁先执行、谁能并行、哪里卡住、哪个节点失败、哪些结果能传给下游。

所以我说 DAG，是因为它适合“编排系统”的底层调度。但从产品语言上，我反而建议不要叫 DAG，除非面向工程师。界面里可以叫：

  • 任务流
  • 编排图
  • 执行计划
  • Agent 工作流
  • 交付流程

“DAG”适合放在技术文档里，不适合放在主界面上吓人。

什么时候用这个工具

我建议看一个判断标准：这个需求是否值得被拆成一个可保存、可调整、可执行、可复盘的计划。

适合用这个工具的场景：

  • 需求比较大，不是一两句话能改完。
  • 需要多个阶段：调研、设计、实现、测试、文档、汇总。
  • 需要多个子 agent 或 skill 分工。
  • 希望先看 AI 的执行方案，再人工调整。
  • 需要中途人工确认。
  • 需要保存过程、产物、日志和最终交付物。
  • 需要可复盘：以后知道当时为什么这么做、哪些节点产出了什么。
  • 要对一个项目持续工作，而不是一次性聊天。

比如这些就适合：

  • “帮我做一个完整的小产品”。
  • “分析这个代码库并重构某个模块”。
  • “规划并执行一套内容生产流程”。
  • “生成一个研究报告，带资料、结构、校验和最终文档”。
  • “让不同 agent 分别做调研、实现、测试、审稿”。

什么时候直接对话

在这里直接和我对话，更适合：

  • 想快速讨论思路。
  • 需求还很模糊，需要一起澄清。
  • 只是一个小修复、小问题、小解释。
  • 想让我快速看代码、改一处、跑测试。
  • 需要高密度来回判断，而不是启动一个流程。
  • 还没确定这件事值不值得编排。

简单说：

对话适合“思考和即时协作”。

这个工具适合“把复杂工作变成可执行系统”。

直接对话

我是一个连续协作的人，灵活、快、可以随时拐弯。适合探索、判断、小修、小范围实现和高密度来回。

编排工具

AI 更像一组被编排的执行单元，有计划、有节点、有日志、有产物、有状态。适合复杂、多阶段、需要留痕的交付。

建议

  • 小事、探索、策略、快速代码修复：直接在这里聊，让我做。
  • 大事、长期任务、多步骤交付、需要留痕：用编排工具。
  • 不确定该不该用工具：先在这里问我，我帮你判断成“直接做”还是“进入编排”。

产品命名建议：界面使用“任务编排图”或“执行计划”，DAG 留在 README 和架构说明里。
