Local-first agent workflow

把一次性对话，变成可编辑、可执行、可复盘的 Agent 工作流。

这个项目是一个本地可运行的 Agent 工作流设计与执行平台：用户输入目标，系统先生成 DAG 任务图；人可以在画布上改节点、依赖、模型和 skill；确认后后端按依赖调度 Codex 执行，并把日志、产物、会话记录沉淀下来。

01 · Positioning

它不是「多开几个聊天框」，而是给 Agent 工作加上工程化骨架。

项目 README 里有一句很关键：Agent View 的核心不是聊天窗口变多，而是把多个后台 agent 的状态、输入需求和生命周期集中到一个可操作视图里。

一句话定位：用可视化 DAG 管住复杂任务，让 Codex 从「回答者」变成「可调度的执行节点」。

分享时可以这样开场

对话适合轻量协作，但复杂任务需要计划、并行、检查点和复盘。这个工具做的事，就是把这些工程化能力包在一个本地网页工作台里。

02 · Usage

怎么用：启动、生成编排、人工调整、确认执行。

项目没有前端打包流程，Node 20+ 即可运行。真实执行依赖本机 Codex CLI；如果只想演示界面和调度，可以开 mock 模式。

启动服务

npm start

打开 http://localhost:8787，进入可视化编排工作台。

演示模式

USE_MOCK_CODEX=1 npm start

不消耗真实 Codex 调用，节点会输出模拟日志，适合产品演示和流程讲解。

运行测试

npm test

覆盖 planner、runner、config/session、skills、weather 和 folder picker 等核心逻辑。

03 · Core Flow

核心思路：先把目标变成结构化计划，再让执行器按图推进。

这里最重要的设计选择，是把「复杂需求」拆成 schema 约束的节点和边。计划可以被人看懂、编辑、保存，也可以被机器调度。

01 输入目标 用户在左侧输入想完成的任务，并选择规划模型与推理强度。
02 生成计划 codex exec 按 JSON Schema 输出 DAG 编排，失败时 fallback 到默认草案。
03 人工编辑 在画布中拖动节点，修改任务、依赖、skill、sandbox、模型和验收标准。
04 依赖调度 Runner 找出依赖已完成的节点，在最大并发限制内启动 Codex 子进程。
05 过程可见 SSE 推送节点状态、stdout/stderr 日志、等待确认和最终结果。
06 沉淀产物 Session 保存 conversation、plan、run 输出，artifact 目录保存用户可交付文件。
04 · Architecture

架构不复杂，但边界清楚：前端管操作感，后端管计划与执行。

代码规模大约 5,400 行，其中前端交互和样式占较大比例；后端模块按配置、规划、运行、会话、skill 发现分层。

HTTP API

server/index.js 提供静态资源、配置读写、计划生成、运行创建、SSE 事件、人工继续、停止运行等接口。

Codex CLI Adapter

server/codexRunner.js 封装 codex exec，支持 sandbox、模型、推理强度、schema 输出、超时和取消。

Local-first Storage

默认把系统记录放在 .orchestrator/sessions，把用户交付物放在 artifacts/<session-id>。

server/planner.js

计划归一化

限制节点数量、去重边、补充 synthesis 节点、套用推理强度偏好。

server/runner.js

DAG 执行器

按依赖和最大并发调度节点，支持 human-review 暂停、继续、取消和事件广播。

server/sessionStore.js

会话记录

创建 session，保存原始计划、当前计划、运行快照、conversation.jsonl 和 manifest。

public/app.js

可视化工作台

负责状态管理、DAG 画布、节点编辑、撤回、缩放、运行面板和人工确认弹窗。

05 · Plan Model

计划模型是这个项目的「契约」：人、前端、后端、Codex 都读同一种结构。

schemas/orchestration-plan.schema.json 约束了计划必须包含 name、summary、strategy、nodes、edges、maxConcurrency 和 finalDeliverable。

nodes 每个节点包含 id、title、agent、task、skills、dependsOn、acceptance、mode、sandbox、坐标等字段。
mode codex 表示可执行节点，human-review 表示人工检查点，synthesis 表示最终汇总。
edges 表达节点依赖关系。Runner 只会启动所有依赖都 completed 的 pending 节点。
maxConcurrency 最大并发限制在 1 到 4 之间，避免本机同时拉起过多 Codex 执行。
acceptance 每个节点都有验收标准，让执行结果不只是「做了」，而是能被后续检查。
06 · Applications

适合所有「值得拆解、执行、验证、沉淀」的复杂任务。

一个简单判断：如果这个需求值得被保存成一张计划图，并且中途可能需要人审、并行、产物归档，它就适合这个工具。

Software

软件开发交付

让不同节点分别负责代码阅读、实现、测试、文档和最终汇总。适合功能开发、重构、缺陷修复和代码审查。

Research

行业研究报告

把资料范围、分主题调研、机会矩阵、人工确认、最终报告拆成可并行节点，过程和出处更容易复盘。

Content

内容生产流水线

选题、资料、草稿、审稿、配图、发布稿可以拆开执行，并在关键节点保留人工判断。

Ops

本地自动化工作台

把重复的工程任务流程化，例如生成文档、整理产物、运行检查、输出报告。

Education

Agent 教学演示

它把「agent 如何拆任务、如何并行、如何等待人确认」可视化出来，非常适合作为教学案例。

Prototype

多 Agent 产品原型

这不是一个完整商业产品，但已经验证了 Agent View、human-in-the-loop、session artifact 的核心交互。

07 · Boundaries

它的价值边界也很清楚：复杂任务更赚，简单对话不必上 DAG。

好工具不是替代所有协作方式，而是把合适的任务变得更可控。这个项目尤其适合阶段多、依赖多、需要复盘的任务。

适合

  • 任务需要调研、实现、测试、文档等多个阶段。
  • 用户想先看计划，再人工调整执行路径。
  • 有多个 skill 或 agent 角色可以协同。
  • 需要保存过程日志、运行结果和最终产物。

不必使用

  • 一句话能回答或一次小改能完成的需求。
  • 需求高度不确定，尚未值得固化成执行图。
  • 安全边界不清楚，却要直接给 workspace-write 权限。
  • 需要长时间云端托管、团队权限和审计的生产场景。

0:00 · 为什么要做它

过去我们和 AI 协作，主要靠对话。对话很自然，但任务一复杂，就会遇到三个问题：拆解不透明、执行状态不可见、过程很难复盘。 这个项目想解决的就是这个问题：把复杂任务从「一串聊天」变成「一张可操作的任务图」。

0:45 · 它是什么

Agent Workflow Design and Execution Platform，中文名 Agent 工作流设计与执行平台，是一个本地可运行的可视化 Agent 编排系统。用户输入需求后，后端调用 Codex 或 Claude Code 生成 DAG 计划； 用户在页面里编辑节点和依赖；确认后由 Runner 按依赖执行，遇到人工确认节点就暂停等待。

1:45 · 怎么用

  • 第一步，npm start 启动服务，打开 localhost:8787。
  • 第二步，输入目标，比如「为当前项目做一个天气查询网页」。
  • 第三步，检查并调整生成的任务流：节点、依赖、skill、模型和推理强度都可以改。
  • 第四步，确认执行。底部会持续显示日志，人工确认节点会弹出确认窗口。

2:45 · 核心机制

机制上分四层：planner 负责把目标变成结构化 JSON；前端负责把 JSON 变成可编辑画布；runner 负责按 DAG 依赖调度； session store 负责把计划、运行、日志和产物保存下来。它的关键不是某个模型，而是把 Agent 执行过程变成可观察、可干预、可复盘的系统。

4:00 · 能用在哪

它适合需要拆解和复盘的任务：软件开发、研究报告、内容生产、多 Agent 教学、本地自动化。反过来，如果只是问一个简单问题，直接对话更快。 这个工具的优势在复杂任务：你能看到谁在做什么、谁依赖谁、哪里卡住、最终产物在哪里。

4:40 · 下一步

下一步可以继续补三类能力：更强的节点模板和权限策略，更完整的运行历史和回放，更好的团队协作与外部系统集成。 但当前版本已经证明了一个方向：Agent 产品不一定只是聊天框，也可以是一个可执行的工作流系统。
